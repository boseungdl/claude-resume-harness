# 이력서 하네스 — 분석 → 초안 → 퇴고

공고별 지원 서류를 만드는 3단계 워크플로. 사실의 원천은 두 파일이며 **직접 수정 금지**다(갱신은 사용자 확인 후에만):
- `about-me/프로필-원천정보.md`(이하 "원천정보") — 커리어·정체성·경력·자격
- `about-me/프로젝트-원천정보.md`(이하 "프로젝트원천") — 프로젝트 사실(역할·스택·성과 라벨·repo 근거). **[BLANK] 표시 항목은 이력서 사용 금지**

## 시작하기 전에

1. 원천정보 최상단 **"LLM 사용 규칙" 섹션을 Read로 로드**하고 이 세션 내내 준수한다. 규칙을 다른 파일로 복사하지 않는다 — 항상 원본을 읽는다.
2. 작업 폴더를 정한다: `about-me/지원/{회사}-{직무}-{YYYYMM}/` (예: `토스-데이터엔지니어-202607`).
3. 사용자가 어느 단계를 원하는지 판별한다:
   - 공고를 붙여넣음 / "공고 분석" → **단계 1**
   - "이력서/자소서 써줘" → `01-분석.md` 존재 확인. **없으면 단계 1로 유도**(임의 진행 금지). 있으면 **단계 2**
   - "퇴고/검토" → `02/03` 파일 존재 확인 후 **단계 3**
4. 원천정보는 77KB다. 본인이든 에이전트든 통째로 읽지 말 것 — 아래에 지정된 섹션만 Grep으로 위치를 찾아 부분 Read한다.

## 단계 1 — 공고 분석

1. JD 원문을 `00-공고.md`로 저장한다. 포함 항목: JD 전문, 회사명·직무명, 마감일, 공고 링크, **자소서 문항과 문항별 글자수 제한**. 문항 정보가 없으면 사용자에게 요청한다. 자소서가 아예 없는 공고면 `00-공고.md`에 "자소서 없음 — 단계 2에서 자소서 skip"을 명시한다.
2. **recruiter 에이전트에 위임 (analyze 모드 지정)**. 프롬프트에 포함할 것:
   - `00-공고.md` 경로
   - 참조 섹션 포인터: 원천정보 §0, §3.5, §3.6, §8, §6-3, §10-A + 프로젝트원천 전체(짧음)
   - 요구 산출: 매칭표(필수/우대×근거×충족도×전략), 회사 리서치 팩트, 시장 벤치마크(유사 공고 3~5건 + 합격 사례), 변형 전략 판정(이 JD에서 앞세울 경험 축 — 직무가 PM·사무직 등 비기술이어도 그에 맞게), 폼 제출 차단 요인
3. **storyteller 에이전트에 위임**. 프롬프트에 포함할 것:
   - `00-공고.md`의 자소서 문항 목록 + recruiter의 변형 전략
   - 참조 섹션 포인터: 원천정보 §★, §1~2, §2-4(장면 아카이브), §3.6, §7 + 프로젝트원천(기술 서사 재료)
   - 요구 산출: 문항별 서사 아크(도입-전개-결론 × 사용 장면 § 출처), 버릴 소재, 소재 공백 리포트
4. 두 결과를 `01-분석.md`로 통합한다. 구성: ① 매칭표 ② 회사 팩트 ③ 시장 벤치마크 ④ 변형 전략 ⑤ 폼 차단 요인 ⑥ 서사 아크. 소재 공백·차단 요인이 있으면 **여기서 사용자에게 보고**하고 답을 받아 반영한 뒤 다음 단계로 간다.

## 단계 2 — 초안 작성 (전제: `01-분석.md` 존재)

1. **`02-이력서.md`**: 변형 전략의 리드 축 순서대로 경험을 배치하고, JD 핵심 키워드가 **상단 1/3**에 노출되게 한다. 파일 최상단에 헤더 블록:
   ```
   <!-- 변형 전략: {리드 축} | 기준 공고: 00-공고.md | 리뷰: 미실시 -->
   ```
2. **`03-자소서.md`**: 서사 아크대로 문항별 작성. 각 문항 제목 옆에 `[제한: N자 / 현재: M자]` 메타 필수(목표 80~100%). 첫 문장은 두괄식. 문항 유형이 `about-me/자소서-마스터.md`(범용 베이스 7문항)와 일치하면 그 베이스에서 변형해 시작한다 — 단 `[회사 커스텀]` 자리는 반드시 이 공고의 분석 결과로 교체.
3. 작성 규칙:
   - 모든 사실은 두 원천에서만 인출. 경력 기간·직무명은 원천정보 §3.5 확정표를, 프로젝트 역할·성과는 프로젝트원천을 그대로 쓴다(수치 라벨 `측정 시`/`예상` 준수, [BLANK] 사용 금지).
   - 서술 문장은 존댓말(~했습니다), 사실 나열 불릿은 개조식.
   - 사실이 담긴 문단마다 근거 주석: `<!-- 근거: §3.6 실무 2차 -->`
   - 완료되지 않은 프로젝트를 완료형으로 쓰지 않는다(§8-5).
4. 초안 완료를 사용자에게 보고한다. **같은 흐름에서 스스로 품질 승인하지 않는다** — 퇴고는 반드시 단계 3의 격리 패스로.

## 단계 3 — 퇴고 (격리된 리뷰 패스)

1. **recruiter(screen 모드)와 editor를 병렬 위임**한다 (한 메시지에 두 Agent 호출):
   - recruiter(screen): `01/02/03` 경로 + 참조 포인터 §3.5, §8-4, §8-5 전달 → 30초 스캔 체크리스트 판정 + 합격 사례 대비 정량 비교표
   - editor: `02/03` 경로 + 문항별 글자수 제한 전달 → 수정 전→후 대조표
2. 두 리뷰를 심각도(치명/권고)로 통합해 사용자에게 보고한다. **치명 항목은 수정안과 함께 제시하고, 사용자 확인 후에만 반영**한다.
3. 반영 후 `02/03` 헤더 블록을 갱신한다:
   ```
   <!-- 리뷰: 제출가능 | 치명 0, 권고 N | YYYY-MM-DD -->
   ```
   치명이 남아 있으면 "제출가능"이라고 쓰지 않는다. 별도 퇴고로그 파일은 만들지 않는다.

## 산출물 구조 요약

```
about-me/지원/{회사}-{직무}-{YYYYMM}/
├── 00-공고.md    # JD 원문 + 문항·글자수 + 마감·링크
├── 01-분석.md    # 매칭표·회사 팩트·벤치마크·변형 전략·차단 요인·서사 아크
├── 02-이력서.md  # 헤더에 변형 전략·리뷰 판정
└── 03-자소서.md  # 문항별 [제한/현재] 메타 + 리뷰 판정
```
